## Unreleased

### Added

- **Handler ↔ scenario coverage gate (per-route BDD-coverage axis, soft-gate landing)** (mokumo#655, quality, Wave 1 hard-gate spine): every `(method, path)` declared in a `Router::route(..)` chain now carries `[happy, error_4xx, error_5xx]` columns counting distinct cucumber scenario names that hit the route with each status class. The fail-closed gate (`scripts/check-handler-scenario-coverage.sh`) requires `happy + 4xx` for new handlers; `5xx` is informational (Posture A in [`docs/adr/adr-handler-scenario-coverage.md`](docs/adr/adr-handler-scenario-coverage.md) — most handlers can't reach 5xx without a fault-injection harness, so requiring it would push every PR into the allowlist). Capture is a tower middleware wired into the `api_bdd` cucumber harness only (`crates/mokumo-shop/tests/api_bdd_world/scenario_coverage.rs`) — production code under `crates/mokumo-shop/src/**` and `crates/kikan/src/**` is untouched. The producer (`tools/docs-gen --bin handler-scenario-coverage`) reuses the syn-based route walker introduced for mokumo#583 (one walker, two consumers) and joins it with the captured JSONL stream into `handler-scenario-coverage.{json,md}`. Allowlist (`<METHOD> <path>  # tracked: <repo>#<n> — <reason>`) and baseline files exempt rows from the gate; a CI-only drift gate (`scripts/check-handler-scenario-allowlist-drift.sh`) fails when a tracked issue closes without removing the matching entry. New CI job `bdd-scenario-coverage` lands in soft-gate posture (`continue-on-error: true`) so the first run can seed `.config/handler-scenario-coverage/baseline.txt`; promotion to a hard gate is two paired edits (drop `continue-on-error`, add to the verdict's `needs:`). Replaces — rather than complements — mokumo#583's per-handler branch-coverage axis: PR #809's experiment showed 71% of mokumo handlers are branchless, making the branch axis structurally blind on this codebase; the route-presence axis is measurable on the actual handler shape. Composes with `scripts/check-route-coverage.sh` (G2 hurl coverage) — both per-route, both fail-closed, but answering different questions (CLI smoke vs acceptance behavior).

- **Scorecard `RowCommon.tool` field for multi-producer support** (#802, quality): the sticky-scorecard wire schema now carries a `tool: String` slug on every row identifying the upstream producer (`coverage-rust`, `crap4rs`, `cargo-mutants`, `bdd-lint`, `bdd-handler-axis`, `gha-wall-clock`, `flaky-scan`, `changed-scope`, `gate-runs`). The renderer surfaces it as a dedicated `Tool` column in the sticky comment so reviewers can tell at a glance which tool emitted each row — the foundation for crap4rs and scrap4rs both contributing rows in a single artifact. Wire-format compatibility: `#[serde(default = "default_tool")]` materializes `"crap4rs"` for any artifact pre-dating this field, and the JSON schema marks `tool` non-required so legacy artifacts validate. Schema is additive; `schema_version` stays at `2`. `inject-check-runs.js` stamps the synthesized `Row::GateRuns` row with `tool: "gate-runs"`. Renderer falls back to the wire-format default when a row arrives without `tool`. Sets up #806 (wire crap4rs v0.4.0 `--format scorecard-row` producer) without coupling to a second mokumo PR.

- **Sticky scorecard V4.1 — per-handler branch-coverage drill-down on `Row::CoverageDelta`** (mokumo#583, quality): the scorecard's coverage row now exposes a collapsible per-handler drill-down listing every HTTP route registered through `Router::route(...)` / `Router::nest(...)` with its branch-coverage percentage. Three independent stages flow into the row: (1) branch-coverage capture via `moon run shop:coverage-branches` (`cargo llvm-cov nextest --lib --branch …`) running against a date-pinned nightly toolchain in `rust-nightly-coverage-toolchain.toml` — branch instrumentation is nightly-only on rustc as of 1.97 ([rust-lang/rust#124137](https://github.com/rust-lang/rust/issues/124137)) and is side-cared via `scripts/nightly-coverage-channel.sh` so the rest of the repo stays on stable; (2) a route-walker producer at `tools/docs-gen --bin coverage-breakouts` that parses the LLVM JSON, walks each crate's HTTP routes with `syn::visit::Visit` to recover `(route, rust_path)` pairs, demangles via `rustc-demangle`, and emits a producer artifact carrying both wire fields and producer-internal diagnostics (unresolved handlers / unresolvable routes / excluded crates); (3) the aggregator's new `--coverage-breakouts-json` flag, which translates to the wire-only `Breakouts { by_crate[], handlers[] }` shape and runs a worst-of handler threshold gate from `quality.toml` (`[rows.coverage_handler]`, defaults: `warn_pct_below = 60.0`, `fail_pct_below = 40.0`, `report_only = true`). The renderer groups handlers under `<details>`-wrapped per-crate sections sorted ascending by `branch_coverage_pct` so the worst-covered handlers surface at the top; a producer-pending note links back to mokumo#583 when no artifact is present. The whole pipeline is **non-blocking by design**: `coverage-handlers` job runs `continue-on-error: true`, the threshold gate is `report_only = true` by default, and the aggregator/renderer fall back gracefully when the artifact is missing. Promotion to a hard gate is two paired edits (remove `continue-on-error: true`, flip `report_only = false`). Removal path when branch coverage stabilizes is a single PR — see `COVERAGE.md#per-handler-branch-coverage-mokumo583`. New schema: `Row::CoverageDelta.breakouts.by_crate[].handlers[]` (additive — `schema_version` stays at `2`); new operator surface: `[rows.coverage_handler]` in `quality.toml`; new CI job: `coverage-handlers` (consumes pinned nightly, uploads `mokumo-coverage-breakouts` artifact).

- **Tracked-exclusion lint (soft-mode landing)** (#740, M00 T1 Wave 1): new `scripts/check-tracked-exclusions.sh` enforces the `tracked: <repo>#<n>` / `adr: <path>.md` annotation contract from `~/.claude/rules/exclusions.md` across four exclusion mechanisms with low false-positive risk — Rust `#[ignore]` and `#[cfg(...skip...)]`, JS/TS `it.skip` / `describe.skip` / `test.skip` / `xtest` / `xit`, and `exclude` / `excluded_files` / `skip` arrays in `.toml`/`.yml`/`.yaml`/`.cjs`/`.js`/`.ts`. The annotation may sit on the candidate line or in the two preceding lines. New `tracked-exclusion-lint` job in `quality.yml` (wired into the `verdict` rollup) and a matching `lefthook` pre-push entry, both invoking the script in `--soft` mode for the T1 baseline period — violations print but don't fail. Promotion to fail-closed lands in a follow-up PR after the baseline corpus is annotated. AGENTS.md §Synchronized-Docs §C documents the contract. The four higher-FP mechanisms (workflow `if:` conditions, commented-out test bodies, TODO/FIXME markers near fixtures, `--exclude <crate>` in CI cargo invocations — already gated by tools-pins) are deferred to T2 AST-aware audit.

- **Sticky scorecard V5 — surface per-gate Check Runs + populate top_failures + two-click rule + fork-PR vitest hardening** (mokumo#770, quality, closes #650): the sticky comment's `Row::GateRuns` row is no longer emitted as a Layer-3 stub. A new injection step in `.github/workflows/scorecard-comment.yml` (between the producer's artifact download and schema validation) calls `octokit.checks.listForRef({ ref: head_sha })` post-Quality-Loop and merges per-gate Check Run data into the scorecard envelope: the top three failing gates populate `top_failures`, and the top five worst-of (any conclusion) populate the `Row::GateRuns` row. The injector (`.github/scripts/scorecard/inject-check-runs.js`) is a pure function with octokit dependency-injected; vitest mocks cleanly. The renderer's two-click rule wraps every row's status indicator in a markdown link to its Check Run URL — rows with a 1:1 binding (`coverage`→`coverage-rust`, `crap_delta`→`crap-delta`, `bdd_feature_skip`/`bdd_scenario_skip`→`bdd-lint`) link directly to the matching gate; the rest fall back to `commit/<head_sha>/checks`. Pending stub rows keep the `⏳` icon but still link somewhere useful. Layer-2 schema gains a defense-in-depth `not: { type: "string", pattern: "<script" }` constraint on every variant's `failure_detail_md` so an XSS payload (from the producer or a fork-PR-supplied path) is rejected at validation. The verdict job is renamed to `Quality Loop (rollup)` so reviewers see at a glance that it's the rolled-up signal; the injector's rollup-name filter handles both old and new names through the rename window. New `.github/workflows/scorecard-e2e.yml` exercises the full producer → injector → validator → renderer pipeline against committed fixtures (no live API) on every push touching scorecard surface area. New vitest cases pin XSS rejection at `/rows/0/failure_detail_md`, the fork-PR head-SHA boundary (URLs reference the fork's HEAD, never the base), and the two-click rule for every row variant. The named "Fork-PR Check Run links resolve against the fork's head commit" BDD scenario lifts off `@future` with matching cucumber-rs step-defs in `tests/bdd_world/fork_pr_steps.rs`. No new Check Run emission code: GitHub Actions' default job→Check-Run mapping already emits per-gate `conclusion: SKIPPED` for `if: false` jobs, verified empirically on prior PRs. See [`QUALITY.md#per-gate-check-runs`](QUALITY.md#per-gate-check-runs) for the full architecture.

- **ADR registry mechanism + `enforced-by:` contract** (#739, M00 T1 Wave 1): every in-repo ADR with YAML frontmatter must declare an `enforced-by:` list-of-objects pairing the decision with a runtime check, lint, dep-absence, workflow, or human-judgment seal. New auto-generated `docs/adr-index.md` (`<!-- AUTO-GEN:adr-index -->` marker) renders the registry table sorted alphabetically by title — empty-set is valid, so the marker pair is load-bearing before any ADR opts in. Three enforcement surfaces: (a) syntactic CI `adr-registry` job — fails any PR touching `docs/adr/*.md` with YAML frontmatter but no `enforced-by:`; (b) reference resolver — `tools/docs-gen/src/validate.rs` library (with the thin `adr-validate` bin shim under `tools/docs-gen/src/bin/`) resolves path-shaped refs (workflows, lint scripts) against the workspace, accepts Rust test path identifiers and clippy lint identifiers (T2 wires `cargo test --list` resolution), runs from `lefthook` pre-push and local dev shells; (c) hand-rolled YAML frontmatter parser in `tools/docs-gen/src/adr.rs` — schema-tight (no new deps), explicit kind enum, descriptive errors. CI stays repo-scoped: the canonical 76-ADR vault lives in the private ops repo and is not checked out by runners; the cross-repo enforcement story is symmetric — ops authors run the same `adr-validate` binary from ops-side `lefthook`. ADRs without YAML frontmatter (legacy format) are dormant under both gates; adoption is voluntary at the file level. Mechanism only — closing #739 also requires the seed-frontmatter sibling PR (8–12 chosen ADRs in `ops/decisions/mokumo/`).

- **Cargo-binary version pinning via `tools.toml`** (#736, M00 T1 Wave 1): new `tools.toml` at the repo root pins versions of every cargo binary on the quality-gate critical path (`cargo-nextest`, `cargo-llvm-cov`, `crap4rs`). Workflow `bins:` lines now use the `name@version` form (`cargo-nextest@0.9.133`, etc.). A new `tools-pins` CI job (and `lefthook` pre-push mirror) runs `scripts/check-tools-pins.sh` which fails on any drift in either direction: pinned-but-unreferenced, referenced-but-unpinned, or version-disagreement. T1 minimum of the Warden pattern from epic #370 council research — NOT the T3 URL+SHA verification, which lives in the deferred-quality umbrella. Ungoverned bins (e.g. `tauri-driver`) pass through silently. The convention is recorded in `AGENTS.md` §"Dep-graph and verdict assertions" alongside the existing dep-graph and verdict rules.

- **Codified-docs Layer 2 paired-files CI gate** (#776, M00): a new `docs-paired-files` CI job (and `lefthook` pre-push mirror) enforces AGENTS.md §Synchronized-Docs §B rules 2 + 3 — every PR adding public crate surface (`pub fn`/`struct`/`trait`/…) under `crates/mokumo-shop/src/**` or any of the 9 kikan satellites (`crates/kikan{,-events,-mail,-scheduler,-socket,-spa-sveltekit,-tauri,-cli,-types}/src/**`) must also touch the matching glossary (`LANGUAGE.md` for the shop vertical, `crates/kikan/LANGUAGE.md` for the platform). Detection is a bash + awk syntax-walk over the PR diff (~60 ms — `cargo public-api` semantic diff was rejected because it requires a nightly toolchain on every Rust-touching PR). Opt-out via the `docs-not-applicable` PR label. Failures post a sticky comment on the PR linking the rule. Wired into the `verdict` aggregator. Semantic rules 1 (trust-boundary code → SECURITY.md) and 4 (architectural change → CONTEXT.md + ARCHITECTURE.md) have no diff signal and are tracked separately under #781.

### Changed

- **Tracked-exclusion lint promoted to fail-closed** (#800, M00 T1 Wave 1, closes parent §B of #740): the 41 baseline exclusion sites surfaced by the soft-mode landing in #798 have been annotated and the `--soft` flag is dropped from both the `tracked-exclusion-lint` job in `quality.yml` and the matching `lefthook` pre-push hook. A PR that introduces an un-annotated exclusion site now blocks the `verdict` rollup; a `git push` doing the same is blocked locally. The dominant cluster (24 entries in `crap4rs.toml` + 6 vite/vitest config entries + 1 `deny.toml` graph exclude) all resolve to one architectural decision documented in [`docs/adr/adr-coverage-exclusions.md`](docs/adr/adr-coverage-exclusions.md), which has been broadened from coverage-only to `Quality-Gate Exclusions for BDD-only, Binary-entrypoint, and Generated Surfaces` — categories 6 (cargo-deny supply-chain graph isolation), 7 (vitest test-runner discovery: cross-runner suites + generated/runner-scope filters), and 8 (barrel-file `src/**/index.ts` coverage exclusions) added. The 4 Playwright `test.skip(process.platform === "win32", ...)` sites carry `tracked: mokumo#416` (Windows runners not yet wired up); the 1 close-to-tray UI placeholder carries `tracked: mokumo#803` (UI not yet implemented). AGENTS.md §Synchronized-Docs §C drops soft-mode language and points at the broadened ADR.

- **`needless_pass_by_value` removed from workspace clippy allow-list** (#795): the final clippy::pedantic cleanup pass — `needless_pass_by_value` is now `warn` (escalated to error by `-D warnings`). 21 sites triaged: 5 flips to `&T` for private helpers (`scorecard::merge_parsed_feature`, `mokumo-shop::admin::diagnostics::build_bundle_sync`, `mokumo-shop::startup::format_db_setup_error`, `kikan::tests::conflict_kind_exhaustive_compile_check`); 16 per-site `#[allow(..., reason = "...")]` annotations for intentional consumes — `*_err` helpers passed to `.map_err(...)` (5 sites across `kikan::activity::repo`, `kikan::auth::repo`, `mokumo-shop::customer::adapter`, `mokumo-shop::shop::adapter`); `Actor::user(impl ToString)` builder ergonomics; CLI `cmd_*` dispatchers with uniform clap-parsed signatures (3 fns in `mokumo-server`); `ConnectionManager::broadcast` for wire-event hand-off. The workspace allow-list now contains only documentation lints (`missing_errors_doc`, `missing_panics_doc`) and judgment-call permanents (`too_many_lines`, `module_name_repetitions`, `struct_excessive_bools`, `struct_field_names`, `items_after_statements`) — clippy::pedantic rollout from #786 is complete.

- **Mechanical clippy lints removed from workspace allow-list** (#786): drained 28 categories from `[workspace.lints.clippy]` — `case_sensitive_file_extension_comparisons`, `default_trait_access`, `duration_suboptimal_units`, `explicit_iter_loop`, `format_push_string`, `ignored_unit_patterns`, `manual_assert`, `manual_let_else`, `map_unwrap_or`, `match_same_arms`, `match_wildcard_for_single_variants`, `needless_continue`, `needless_raw_string_hashes`, `no_effect_underscore_binding`, `option_as_ref_cloned`, `range_plus_one`, `redundant_closure_for_method_calls`, `semicolon_if_nothing_returned`, `similar_names`, `single_match_else`, `trivially_copy_pass_by_ref`, `uninlined_format_args`, `unnecessary_literal_bound`, `unnested_or_patterns`, `unnecessary_wraps`, `unreadable_literal`, `unused_async`, `used_underscore_binding`. Most edits applied by `cargo clippy --fix`; the rest are 1-3 line hand fixes. Per-site `#[allow(..., reason = "...")]` annotations document the few intentional exceptions (axum route handler signatures, `case-sensitive` extension matches against files we generated ourselves, exhaustive compile-time variant checks). `items_after_statements` joins `too_many_lines`, `module_name_repetitions`, `struct_excessive_bools`, and `struct_field_names` as a permanent allow with a justification comment in `Cargo.toml` — the lint is hostile to common test-code patterns where a helper struct or fn is declared inside a `#[test]` near its first use. `needless_pass_by_value` (signature-changing, 54 sites) is deferred to a follow-up. The remaining workspace allow-list now contains only documentation lints + judgment-call permanents.

- **Cast & float-comparison clippy lints removed from workspace allow-list** (#786): `clippy::cast_possible_truncation`, `clippy::cast_possible_wrap`, `clippy::cast_sign_loss`, `clippy::cast_precision_loss`, `clippy::cast_lossless`, and `clippy::float_cmp` now `warn` (escalated to error by `-D warnings` in `moon run shop:lint`). 33 cast sites converted to `T::try_from(x).expect("<invariant>")` where the upstream invariant is documented, or `T::from(x)` where the conversion is lossless. The 28 `float_cmp` sites are all in scorecard test modules deserialising TOML literal values — annotated with one `#[allow(clippy::float_cmp, reason = "...")]` per `mod tests` block (3 modules) since exact equality is the test contract, not a float-arithmetic comparison. Two `cast_precision_loss` sites in display-only byte-formatting helpers carry per-site `#[allow]` with reason. The remaining cleanup categories are still tracked under #786.

- **Clippy `pedantic` enabled at workspace level** (#786): the `pedantic` lint group is now `warn` in `[workspace.lints.clippy]` and inherited by every member crate via `[lints] workspace = true`. The CI lint command (`moon run shop:lint`) already escalates with `-D warnings`, so any new pedantic violation now blocks merge. Lint categories the existing codebase already trips are explicitly allowed in the workspace lint table; the umbrella tracking issue is #786, which lists each allowed category and recommended cleanup order. Goal: tighter AI-agent guardrail against sloppy patterns (cast safety, float comparisons, `let-else`, format-arg inlining, etc.) in new code without a 1755-line cleanup PR.

- **Verdict aggregate gate is now `toJSON(needs) | jq`-driven** (#736, M00 T1 Wave 1): the `verdict` job in `.github/workflows/quality.yml` no longer maintains a parallel `env:` block of `${{ needs.<job>.result }}` entries alongside the `needs:` array. Adding a new upstream gate is now a single-line edit to `needs:` — the assertion code reads `${{ toJSON(needs) }}` once and walks it via `jq`. Same pass/fail semantics: `success` and `skipped` (path-filter skip) pass; `failure`, `cancelled`, and anything else fail. `if: always()` preserved so the verdict still runs when an upstream fails. The new "Dep-graph and verdict assertions" section in `AGENTS.md` codifies this convention alongside the existing `cargo metadata` / `cargo tree` rule for workspace dep-graph checks.

- **Codified-docs Layer 1+2 mechanism (Wave 1 README)** (#741, M00): introduces `<!-- AUTO-GEN:* -->` marker infrastructure for machine-maintained doc sections. The `docs-gen` binary (`tools/docs-gen`) renders the MSRV badge in `README.md` from `Cargo.toml`'s `workspace.package.rust-version`; `moon run docs:gen` wraps it; a new `docs-drift` CI job enforces no-drift on every PR via `git diff --exit-code`. The Synchronized-Docs rule in `AGENTS.md` declares all source→target relationships and points at the registry in `tools/docs-gen/src/registry.rs` where new sections plug in. Pre-push hook fires when `Cargo.toml`, `README.md`, or `tools/docs-gen/**` changes. Foundation for Group A+B doc automation (#743, #744, #745, QUALITY, COVERAGE).

- **Auth HTTP handlers promoted to kikan platform layer** (#685, M00):
  - `POST /api/platform/v1/auth/{login, logout, me, recover/request, recover/complete}` are now the canonical kikan-served URLs, served by adapters in `kikan::platform::v1::auth` over the pure-fn core in `kikan::control_plane::auth`. Legacy `/api/auth/*` URLs (including `/forgot-password`, `/reset-password`, `/recover` recovery-code variant) continue to route to the same handlers through M0 so the shop SPA is unchanged.
  - **Recovery-session redesign (Recovery Session Token pattern):** the storage key for a pending password reset is now an opaque 256-bit hex `RecoverySessionId` minted at issuance, not the user's email. The user-facing PIN remains the secret. Anti-enumeration: "unknown email", "wrong PIN", "expired session", and "attempts exhausted" all emit the same `400 Invalid or expired recovery session` body. TOCTOU-safe attempts counter via atomic `DashMap::remove → check → conditional reinsert`; two-tier eviction (lazy on every redeem + 60s background sweep cancelled on shutdown).
  - `BootConfig::with_recovery_writer` runtime hook lets verticals own where their recovery artifact lands (file, email, push, …) without leaking branding into kikan. Mokumo wires the existing branded HTML writer at the hash-derived path.
  - kikan-admin-ui login form retargeted from the dead `/api/platform/v1/auth/sign-in` URL to the canonical `/api/platform/v1/auth/login`.
  - In-PR fix: legacy `forgot-password` now runs the recovery-rate-limit gate before issuing a session (the pre-promotion handler skipped it).
  - Closes #271 (cross-crate `SeaOrmUserRepo::new` anti-pattern dissolves naturally — handlers now live in the same crate as the repo). F1 follow-up #698 tracks shop-SPA retargeting `/api/auth/*` → canonical URLs.

### Fixed

- **`adr-validate` tolerates unknown YAML block-sequence/mapping keys in ADR frontmatter** (#793): the hand-rolled parser in `tools/docs-gen/src/adr.rs` previously bailed on any indented continuation line under an unknown top-level key (e.g. `related:` followed by `  - decisions/foo.md`). It now skips past indented and blank/comment lines that belong to an unknown key with an empty inline value, so cross-link frontmatter parses cleanly. The strict schema for `enforced-by:` is unchanged — its block-sequence form still routes through `apply_enforced_by_opener` and rejects malformed items as before. Unblocks the cross-repo `adr-validate --adr-root ~/Github/ops/decisions/mokumo` run against the canonical ADR vault (68 ADRs, all references resolve).

- **Sticky scorecard `crap-rust` table no longer surfaces false 0% coverage rows** (#783): `cargo-llvm-cov nextest` (without `--lib`) dual-compiled each source file — once via the lib's `#[cfg(test)] mod tests`, once via each integration-test binary that depended on it — and emitted two `FN`/`FNDA` records per function with build-hash-prefixed mangled names. `crap4rs` keys off function name and picked the lower (zero-hit) record per name, manufacturing fake 0% coverage rows on PRs whose code was actually well-covered. The fix adds a small `lcov-dedup` step between `cargo llvm-cov` and `crap4rs` (lives in `tools/docs-gen` alongside `adr-validate`) that collapses duplicate `FN`/`FNDA` records by `(file, line)` summing hit counts, then recomputes `FNF`/`FNH` totals from the deduplicated set. `--lib` (the simpler fix) was rejected because non-BDD integration tests under `crates/kikan/tests/` (e.g. `dag_resolver`, `engine_integration`, `m00_platform_migrations`, `migration_conn`) exercise lib code that is in-scope for CRAP and provide legitimate coverage signal — `--lib` would silently drop their contribution and trip CRAP on functions that are actually covered.

- **Insta + cargo-mutants tmp-tree resolution** (#736, M00 T1 Wave 1): new `crates/mokumo-shop/build.rs` bakes `INSTA_WORKSPACE_ROOT` into the test binary at compile time via `cargo:rustc-env`. `insta` resolves snapshot paths through `option_env!("INSTA_WORKSPACE_ROOT")` (`Workspace::UseAsIs` short-circuit) and never invokes `cargo metadata` at runtime. Required for `cargo-mutants`, whose reflinked tmp tree breaks `cargo metadata` resolution and previously forced `--baseline skip`. `mokumo-shop` is the only insta consumer in the workspace; its crate location at `<workspace>/crates/mokumo-shop/` makes `CARGO_MANIFEST_DIR/../..` the workspace root, with a compile-time `assert!(workspace_root.join("Cargo.toml").exists())` catching any depth-bug at build time. Wires the V6 mutation gate (#748) consumer; this PR ships the mechanism, not the gate.
- **Bootstrap admin path now marks setup-complete** (#725, kikan): `bootstrap_admin_with_codes` (used by `mokumo-server bootstrap`) was missing the `settings.setup_complete='true'` write that `create_admin_with_setup` (HTTP setup wizard) performs, so the two first-admin paths left the install in different states. Both now call a shared `mark_setup_complete` helper inside their existing transactions, keeping the admin row and the settings row atomic.
- **Legacy install upgrade migrates `users`/`roles` to meta.db with crash-recovery semantics** (#705, kikan): pre-PR-A installs (binaries from before mokumo#679) carry `users` and `roles` tables on the per-profile DB; the post-PR-A schema places them on `meta.db`. The legacy upgrade now (1) extends `kikan::db::check_schema_compatibility` to recognise platform-owned migration rows so the binary stops rejecting legitimate legacy fixtures as "newer-version" databases, and (2) runs a state-machine data move ahead of the existing `meta.profiles` insert that observes physical state on each boot — fresh upgrade (insert + drop), crash recovery (drop only when prior commit reached meta but not the per-profile drop), or no-op when the upgrade already completed. TOCTOU is closed via a SeaORM transaction whose first statement upserts `meta.legacy_upgrade_locks` (new platform migration) to force `RESERVED` immediately. Multi-legacy-profile collisions on user IDs or custom role IDs (id > 3) are detected pre-mutation and abort with a privilege-escalation diagnostic; partial email intersections (indicating external mutation of meta.db) abort as anomalous. 22 new tests across `kikan/src/meta/upgrade.rs` (13) and `kikan/tests/features/legacy_upgrade.feature` (9 scenarios).
- **Shop logo 404 now returns vertical error code** (`mokumo-shop`): `GET /api/shop/logo` and `DELETE /api/shop/logo` when no logo is set now return `{"code":"shop_logo_not_found","..."}` instead of the platform-wide `{"code":"not_found","..."}`. Fixes #695 — the frontend, BDD spec, and Hurl smoke test all expected the vertical code.
