## Unreleased

### Added

- **Engine/UI version-mismatch detection** (#502): kikan grows an unauthenticated `GET /api/kikan-version` endpoint (`kikan::data_plane::kikan_version::kikan_version_router`) returning `{ api_version, engine_version, engine_commit, schema_versions }`. `api_version` is the semver of the HTTP API contract (new `kikan_types::API_VERSION = "1.0.0"`); `engine_version` is `kikan`'s crate version; `engine_commit` is the short git SHA baked in via `crates/kikan/build.rs`; `schema_versions` is a `{ profile_dir_name: highest_applied_migration }` map sourced from each pool's `seaql_migrations` table. Verticals mount the router alongside `/api/health`; mokumo-shop does this in `data_plane_routes`. The admin SPA gains a build-time constant `__KIKAN_ADMIN_UI_BUILT_FOR__` (Vite `define`), fetches `/api/kikan-version` on layout mount (outside the Tauri-only block so `mokumo-server --spa-dir` and Cloudflare-tunnel deployments are covered), and renders a non-blocking `<VersionMismatchBanner />` on mismatch. A vitest drift guard pins `KIKAN_ADMIN_UI_BUILT_FOR` to `kikan_types::API_VERSION`; Playwright BDD covers the mismatch and match paths by intercepting `/api/kikan-version` with `page.route()`.

- **Feature-gated SPA primitive: `kikan::data_plane::spa::SpaSource` + sister crate `kikan-spa-sveltekit`** (#499): kikan grows a `SpaSource` trait â€” a domain-neutral seam for mounting a single-page-app as the data-plane fallback â€” and `Graft` grows an optional `fn spa_source() -> Option<Box<dyn SpaSource>>` hook (default `None`). `Engine::new_with` captures the source once at construction (so `build_router` can consume it without reaching back through `PhantomData<G>`); `build_router` installs it via `fallback_service` before the middleware stack, so API routes register first and the SPA inherits every layer (auth, CSRF, security headers, tracing). When an SPA is mounted, the engine also registers typed JSON-404 catch-alls at `/api` and `/api/{*rest}` so unmatched API paths continue to return the API error contract rather than the HTML shell. The actual embedding strategies live in the new sister crate `kikan-spa-sveltekit`, which ships two `SpaSource` impls: `SvelteKitSpa<A: rust_embed::RustEmbed>` for single-binary builds, and `SvelteKitSpaDir { dir: PathBuf }` for on-disk layouts (backed by `tower_http::ServeDir` + `ServeFile` + a response middleware that stamps SvelteKit's cache policy: 1-year immutable under `_app/immutable/*`, `no-cache` on any HTML response including missing-asset fallbacks to the shell, `no-store` on non-2xx responses, 1-hour on everything else). `kikan` itself stays `rust-embed`-free â€” invariant I5 holds because the sister-crate pattern keeps build-artifact dependencies at the consumer's edge. `mokumo-desktop` moves its `#[derive(rust_embed::Embed)] struct SpaAssets` out of the deleted `crates/mokumo-spa` into its own `src/lib.rs` and injects a `SvelteKitSpa<SpaAssets>` via `MokumoApp::with_spa_source(â€¦)`. `mokumo-server` gains `--spa-dir <PATH>`: when present, the binary boot-validates ``<PATH>/index.html`` (exact error: ``spa directory <PATH> has no index.html â€” did you run `pnpm build`?``, exit 2) and injects a `SvelteKitSpaDir`; when absent, the server logs ``mokumo-server starting in API-only mode (no --spa-dir provided)`` at info level and non-API paths return Axum's default 404. `crates/mokumo-spa` is deleted wholesale (no deprecation shim â€” internal-only crate). See `ops/decisions/mokumo/adr-kikan-primitive-crate-design.md` and the discover+shape cycle at `ops/workspace/mokumo/20260422-499-spa-primitive-prebuild/`.

- **`DataPlaneConfig` + 3-state `DeploymentMode`** (#500): kikan grows a `data_plane` module owning the HTTP middleware stack. `DeploymentMode` retires the 2-state `{Lan, Loopback}` placeholder in favor of `{Lan, Internet, ReverseProxy}` per `adr-kikan-deployment-modes`. `DataPlaneConfig` carries `bind_addr`, `allowed_origins` (`Vec<HeaderValue>`), `allowed_hosts` (`Vec<HostPattern>`), and the selected mode; `BootConfig::with_data_plane(cfg)` and `with_deployment_mode` / `with_bind_addr` shortcuts feed it into the engine. The middleware stack now consults the mode: session cookies switch between `SameSite=Lax`/`Secure=false` (Lan) and `SameSite=Strict`/`Secure=true` (Internet / ReverseProxy); a custom double-submit CSRF layer gates non-`GET` requests in non-Lan modes with an `Origin`/`Referer` allowlist; a per-IP global sliding-window rate limiter engages in non-Lan modes; `X-Forwarded-For` / `X-Forwarded-Proto` are trusted only under `ReverseProxy` and stripped otherwise. mDNS service registration is gated on `DeploymentMode::mdns_enabled`. `HostPattern` is a validated newtype that rejects empty, whitespace, NUL, path separators, port, scheme, wildcards, and non-ASCII at parse time (ADR 2026-04-22 (a) pattern). `mokumo-server` replaces `--mode lan|loopback` with `--deployment-mode lan|internet|reverse-proxy` plus `--allowed-host`, `--allowed-origin`, and `--host` overrides; non-Lan modes require at least one allowed host and origin.

- **`Graft` capability/vocabulary split** (#617, #618): The `Graft` trait grows `type ProfileKind` plus five vocabulary hooks â€” `db_filename`, `all_profile_kinds`, `default_profile_kind`, `requires_setup_wizard`, `auth_profile_kind`. Kikan no longer names concrete profile variants (`SetupMode`) or Mokumo's DB filename (`mokumo.db`) anywhere in production code; every yes/no about a profile is asked through the vertical via a hook. `PlatformState` is restructured to hold an `Arc<HashMap<ProfileDirName, DatabaseConnection>>` pool map plus a `ProfileDirName` opaque newtype at the API boundary. Auth types (`AuthenticatedUser<K>`, `ProfileUserId<K>`, `Backend<K>`, `ActiveProfile<K>`, `ProfileId<K>`, `switch_profile<K>`, `verify_credentials<K>`, `switch_profile_admin<K>`, and the profile-routing middleware) are all generic over the graft's `ProfileKind`. The `Display`+`FromStr` pair on `ProfileKind` is the single source of truth for on-disk directory names; `Engine::boot` validates the round-trip at startup and refuses to boot a graft whose `Display`/`FromStr` are not inverses. `ProfileDirName` now validates on construction â€” rejects empty names, `.`, `..`, path separators, NUL, and leading dots, closing a path-traversal vector at the API boundary. See `adr-kikan-engine-vocabulary`.

- **Data-plane router composition + `try_bind` lift to kikan** (#501): The eight-layer middleware composition that wraps each graft's data-plane routes moves out of `Engine::build_router` into a new `kikan::data_plane::router` module. `compose_router(ComposeInputs<S, K>) -> Router` owns the layer-order contract (host allowlist â†’ forwarded â†’ rate limit â†’ security headers â†’ trace â†’ auth â†’ CSRF â†’ profile DB), the `/api/**` typed-JSON-404 catch-all, and the optional SPA fallback. `Engine::build_router(&self, state)` becomes a thin orchestrator that builds the inputs struct and delegates â€” public signature unchanged. The port-range bind helper lifts out of `mokumo_shop::startup::try_bind` into `kikan::data_plane::bind::try_bind`; mokumo-shop loses the function and `apps/mokumo-server` (plus three integration tests) import the new path. No public-API changes; no behavior changes; closes the `kikan-extraction-finalization` pipeline.

- **`Graft` recovery + bootstrap vocabulary split** (#619, #620): The `Graft` trait grows three new hooks so the vertical owns its own reset + bootstrap vocabulary: `recovery_dir(&profile_id) -> Option<PathBuf>`, `setup_token_source() -> SetupTokenSource`, `valid_reset_pin_ids() -> &'static [PinId]`. Each hook ships with an empty/disabled default; verticals that don't use a file-drop recovery flow or a setup-wizard token compile unchanged. Kikan-owned supporting types land in `kikan::control_plane::types`: the `SetupTokenSource` enum (`Disabled` / `File(PathBuf)` / `Inline(Arc<str>)`) and the `PinId(Arc<str>)` opaque newtype. `PinId` validates on construction â€” rejects empty, whitespace-only, or NUL-containing inputs â€” following the `ProfileDirName` precedent. See `adr-kikan-engine-vocabulary` Â§ "Amendment 2026-04-22 (b)".

### Changed

- **`DataPlaneConfig` construction is now validated at the type boundary** (#635): `kikan::DataPlaneConfig` gains `fn new(mode, bind_addr, allowed_hosts, allowed_origins) -> Result<Self, ConfigError>` that rejects non-Lan modes (`Internet`, `ReverseProxy`) with an empty `allowed_hosts` or `allowed_origins` list, returning the new `ConfigError::EmptyAllowedHosts { mode }` / `EmptyAllowedOrigins { mode }` variants. The constructor also ASCII-lowercases every `allowed_origins` entry so the CSRF double-submit layer's byte-exact compare against the browser's already-lowercase `Origin` header still matches when a caller passes `HTTPS://Shop.Example.COM`. A private `_priv: ()` sentinel field blocks struct-literal construction from outside the `config` module so these invariants cannot be bypassed; `lan_default` is unchanged for LAN-posture callers. The `apps/mokumo-server` CLI drops both its pre-construction host/origin empty-check AND its redundant origin lowercase step (the type boundary now owns them) and prints the structured error before exiting with code `2`. No observable runtime behavior change on the supported paths â€” a library caller (e.g., a future programmatic embedder of `kikan`) can no longer boot a non-Lan server that 400s every POST.

- **Mokumo admin surface hoisted to `mokumo-shop`** (#617, #618): Diagnostics (`GET /api/diagnostics`, `GET /api/diagnostics/bundle`), profile listing, the admin UDS router (`kikan::Engine::admin_router` + `kikan::admin::build_admin_router`), the wire-DTO `switch_profile_admin` wrapper, the demo-reset handler (`POST /api/demo/reset`), the auth HTTP handlers (`/api/auth/*`, `/api/setup`, account recovery, demo auto-login middleware), the user-admin HTTP handlers (`/api/users/*`), migration status (`collect_migration_status`, `GET /migrate/status` UDS + CLI), and backup status (`GET /api/backup-status` + `GET /backups` UDS) all move from `kikan::{platform,control_plane,engine::admin}` to `mokumo-shop::{admin,demo_reset,auth_handlers,user_admin}`. `kikan::platform_public_routes()` is removed â€” kikan no longer carries any routes that name the vertical's profile dirs. Wire shapes unchanged with one observable value: `DiagnosticsResponse.app.name` in `GET /api/diagnostics` + the admin UDS now reports `"mokumo-shop"` instead of `"kikan"` â€” the field is `env!("CARGO_PKG_NAME")` in the crate owning the snapshot fn. `build_admin_router` now lives at `mokumo_shop::admin::build_admin_router`; `apps/mokumo-server` instantiates it directly. The `kikan::tenancy::layout::migrate_flat_layout` dead duplicate is deleted â€” the live copy has always been in `mokumo_shop::startup::migrate_flat_layout`.

- **`ControlPlaneState` shrinks; Mokumo absorbs recovery + reset-PIN vocabulary** (#619, #620): `reset_pins: Arc<DashMap<String, PendingReset>>` and `recovery_dir: PathBuf` lift off `kikan::ControlPlaneState` into `mokumo_shop::state::MokumoShopState`. The `PendingReset` struct moves from `kikan::control_plane` to `mokumo_shop::auth::PendingReset`. `ControlPlaneState.setup_token` changes type from `Option<String>` to `Option<Arc<str>>` and its derivation path moves off `Engine::boot` parameters onto `Graft::setup_token_source()` â€” the engine resolves the source once at boot (`File` reads fail-fast with `EngineError::Boot`). `Engine::boot`'s signature drops two parameters (`setup_token: Option<String>`, `recovery_dir: PathBuf`). `MokumoApp` stops being a unit struct â€” it gains `MokumoApp::new(setup_token)` and `.with_recovery_dir(path)` for tests and managed deployments. The file-drop reset HTTP handlers (`/api/auth/forgot-password`, `/api/auth/reset-password`) move to a `reset_router()` mounted on `SharedMokumoState` (was on `ControlPlaneState`) so the routes can reach the Mokumo-side reset-PIN map and recovery directory. Wire shapes unchanged â€” the reset + setup BDD scenarios pass without modification.

- **`Graft::on_post_reset_db` signature** (#619): drops the `_recovery_dir: &Path` parameter. Verticals that need a recovery directory resolve it internally (Mokumo via `self.effective_recovery_dir()`). The `mokumo-recovery-*.html` cleanup pattern moves from `kikan-cli::reset_db::run` into `mokumo_shop::lifecycle::cleanup_recovery_files`, closing the last kikan-cli-side Mokumo-vocabulary leak. `kikan_cli::reset_db::run` drops the `recovery_dir: &Path` parameter and now derives the DB filename from `Graft::db_filename()`; `ResetReport::recovery_dir_error` is removed (dead field after the lift).

- **I1 strict-pattern CI check** (#617, #618, #619): `scripts/check-i1-domain-purity.sh` gains a second pattern class flagging `\bSetupMode\b`, `"mokumo\.db"`, and `\bPendingReset\b` under `crates/kikan/src/` (excluding `*_tests.rs` / `**/tests.rs`). "Classic I1" stops garment nouns; "strict I1" stops the vertical's wire artifacts and reset-entry shape from leaking back into kikan's production code.

### Performance

- **Composite index on `activity_log(created_at DESC, id DESC)`**: Eliminates filesort on paginated activity-log reads. `created_at` is stored at second precision, so batch inserts within the same second produce identical timestamps; the composite index gives the `ORDER BY created_at DESC, id DESC` tie-breaker an O(log n) path. (#567)

### Security

- **Login timing side-channel closed**: `verify_credentials` now runs argon2 against a reference hash on the unknown-email and inactive-account paths, so response time no longer reveals whether an email is registered. (#508)
- **Inactive accounts can no longer regenerate recovery codes**: `regenerate_recovery_codes` rejects inactive accounts before password verification. (#508)

### Fixed

- **`platform_bdd` suite wired into CI + step defs run platform migrations first** (#647): The `crates/mokumo-shop/tests/platform_bdd` harness was passing locally but orphaned from `moon.yml` and `.github/workflows/quality.yml`, so regressions in platform-schema integration could reach `main` unseen. Adds a `shop:test-platform-bdd` moon task and appends it to the `BDD acceptance tests` CI step. While wiring the gate, a latent bug surfaced: two step definitions in `migration_safety_steps.rs` (`given_database_at_schema_version`, `given_db_at_schema_version`) connected a fresh DB and ran only the vertical `Migrator::up`, skipping `kikan::migrations::platform::run_platform_migrations`. Scenarios with low version counts passed by coincidence; "Only the last three backups are kept" failed with `no such table: users` once `Some(5)` reached the `login_lockout` migration. Both steps now run platform migrations before the vertical migrator, mirroring the production ordering in `mokumo_shop::db::initialize_database`.
- **Diagnostics surfaces real I/O errors**: `read_profile_diagnostics` now propagates `tokio::fs::metadata` errors instead of swallowing them; only `NotFound` continues to map to absent file size. (#508)
- **Diagnostics bundle no longer blocks the async executor or grows unbounded**: `build_bundle` and `collect_system_diagnostics` now run on the blocking thread pool. Bundle log inclusion is capped at `MAX_LOG_FILES=32` and `MAX_LOG_TOTAL_BYTES=64 MiB` (raw-byte cap enforced at read-time so a single huge line cannot exhaust memory); when the cap trips, a `logs/TRUNCATED` marker entry is written so the operator knows data was dropped. (#508)
