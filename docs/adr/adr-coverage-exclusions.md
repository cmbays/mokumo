# ADR: Quality-Gate Exclusions for BDD-only, Binary-entrypoint, and Generated Surfaces

**Status**: Accepted
**Date**: 2026-04-26
**Updated**: 2026-05-04 (PR #800 — broadened scope to cargo-deny supply-chain graph and vitest test-runner discovery; added barrel-file category)
**PR**: #692, #800

## Context

Mokumo's quality gates — coverage (`cargo-llvm-cov`), CRAP score
(`crap4rs`), supply-chain audit (`cargo-deny`), and frontend test
runner (`vitest`) — each define an `exclude` list of paths or crates
they should not traverse. Several categories of exclusion are
permanent design choices, not deferred work: their exclusion will
never resolve into inclusion. Documenting all of them in one place
keeps the rationale grep-able and prevents future maintainers from
re-litigating each one.

The eight categories below are either exercised by integration
harnesses that don't emit the gate's signal, owned by a different
runner, or generated/build artifacts. If we let any flow through the
gate that owns them, the threshold trips on permanently-uncovered
code, gate noise drowns out real regressions, and the team learns to
ignore the gate. Excluding them is the right answer — the question
is **how** to track that exclusion.

1. **Cucumber-rs BDD harnesses** — `crates/mokumo-shop/tests/api_bdd/`,
   `crates/mokumo-shop/tests/admin_uds.rs`, etc. The `cucumber` crate
   uses a custom test harness incompatible with `nextest`, and its
   scenario steps don't produce per-line coverage in the harness output.
   Handlers driven exclusively through these harnesses (e.g., kikan
   platform/control-plane, mokumo-shop's HTTP adapters and admin UDS
   handlers) therefore read as 0% covered, even though semantic
   coverage through the BDD scenarios is high.

2. **Binary entry points** — `apps/mokumo-server/`, `apps/mokumo-desktop/`,
   `crates/mokumo-shop/src/{startup,cli}.rs`. These have no unit-test
   surface; their behavior is verified by `demo-smoke`, `api-smoke`,
   `desktop-e2e` integration jobs, none of which emit LCOV.

3. **Tauri-specific crates** — `kikan-tauri`, `mokumo-desktop`. Tauri's
   runtime requires a desktop event loop; `cargo nextest` can't host
   one, so these crates can't be tested under llvm-cov at all. They
   get their own desktop-e2e job.

4. **Frontend-bundling crate** — `kikan-admin-ui`. Uses `rust-embed`
   with `#[folder = "frontend/build"]` — a *compile-time* check that
   the SvelteKit build output exists. `cargo nextest`-based coverage
   doesn't run `pnpm build` first, so the crate fails to compile in
   the coverage pipeline.

5. **Test/build helpers** — `**/tests/**`, `**/*_tests.rs`,
   `**/build.rs`. By definition either zero coverage (test fixtures
   are the test input) or zero meaningful CRAP score (build scripts).

6. **Supply-chain graph isolation** (cargo-deny). `mokumo-desktop`
   pulls Tauri's platform-specific system deps (`cocoa-foundation`,
   `winapi`, etc.) whose advisories vary by host OS. cargo-deny's
   `[graph].exclude` keeps the workspace-wide advisory + license
   check deterministic and host-OS-agnostic. Tauri-specific
   advisories are audited separately in the `desktop-e2e` job, where
   the right host-OS baseline is in scope.

7. **Vitest test-runner discovery.** Two sub-rationales share this
   category:
   - **Cross-runner suites.** `apps/web/tests/smoke/**` is owned by
     Playwright, not vitest. Excluding it from vitest's discovery
     prevents vitest from attempting to execute Playwright-shaped
     specs.
   - **Generated and runner-scope filters.** `.features-gen/**`
     (generated by `bdd-lint` from `.feature` sources),
     `tests/demo-captures/**` (Playwright recording artifacts),
     `**/.claude/**` (agent metadata), `.svelte-kit/**` (SvelteKit
     codegen), `build/**` (build output), `tests/features/**` and
     `tests/screens/**` (BDD scenarios + screen captures). These
     paths are generated, build artifacts, or non-vitest content;
     traversing them produces noise without signal.

8. **Barrel-file coverage exclusions.** `src/**/index.ts` (and
   `src/index.ts` in single-binary tooling crates such as
   `tools/bdd-lint`) are re-export aggregators containing only
   `export { ... } from "./..."` lines — no executable logic.
   Including them in v8 coverage dilutes meaningful coverage signal
   with always-100% re-export lines.

## Decision

These exclusions are **architectural and permanent**. They are
documented here once; the config sites reference this ADR rather than
carrying ad-hoc justifications.

The annotation contract from the global rule and the in-repo
`tracked-exclusion-lint` (CI gate, see `AGENTS.md` §Synchronized-Docs
§C) requires the literal `adr: <path>.md` keyword form on the same
line as each exclusion entry or within the two preceding lines. A
cluster header carries the keyword once
(`# adr: docs/adr/adr-coverage-exclusions.md — <short description>`);
long consecutive runs need additional bare keyword lines every
two-to-three entries to satisfy the lint's regex window.

The exclusions live in the gate-config files below.

### 1. `crap4rs.toml` — CRAP score gate (categories 1–5)

```toml
preset = "strict"

exclude = [
  # adr: docs/adr/adr-coverage-exclusions.md — BDD-only platform handlers
  "crates/kikan/src/platform/**",
  # adr: docs/adr/adr-coverage-exclusions.md — BDD-only control-plane handlers
  "crates/kikan/src/control_plane/**",
  # adr: docs/adr/adr-coverage-exclusions.md — BDD-only HTTP adapters (api_bdd)
  "crates/mokumo-shop/src/restore_handler.rs",
  "crates/mokumo-shop/src/profile_switch.rs",
  # adr: docs/adr/adr-coverage-exclusions.md
  "crates/mokumo-shop/src/routes.rs",
  "crates/mokumo-shop/src/setup.rs",
  # adr: docs/adr/adr-coverage-exclusions.md
  "crates/mokumo-shop/src/settings.rs",
  # ... mid-cluster `# adr:` lines repeat every 2–3 entries to satisfy the lint window ...
  "**/build.rs",
]
```

The cluster header carries the rationale; the bare mid-cluster
`# adr:` lines exist so the regex floor in
`scripts/check-tracked-exclusions.sh` accepts every entry. See the
file itself for the full set.

### 2. `crates/mokumo-shop/moon.yml` — coverage and clippy commands (categories 1–5)

```yaml
# All cargo invocations carry the same --exclude trio. Rationale: see
# docs/adr/adr-coverage-exclusions.md
test:
  command: cargo nextest run --profile ci --workspace --exclude mokumo-desktop --exclude kikan-tauri --exclude kikan-admin-ui
crap:
  command: |
    cargo llvm-cov nextest --workspace --exclude mokumo-desktop --exclude kikan-tauri --exclude kikan-admin-ui --lcov ...
# (and coverage-rust, coverage-html, lint-rust)
```

The CI workflow's `crap-scorecard` job mirrors the same exclude trio
inline.

### 3. `deny.toml` — supply-chain graph (category 6)

```toml
[graph]
# adr: docs/adr/adr-coverage-exclusions.md — Tauri-specific deps live in mokumo-desktop only
exclude = ["mokumo-desktop"]
```

### 4. `apps/web/vite.config.ts`, `crates/kikan-admin-ui/frontend/vite.config.ts`, `tools/bdd-lint/vitest.config.ts` — vitest test discovery (categories 7, 8)

```typescript
test: {
  exclude: [
    ...configDefaults.exclude,
    // adr: docs/adr/adr-coverage-exclusions.md — Playwright-owned smoke suite (cross-runner)
    "tests/smoke/**",
    // adr: docs/adr/adr-coverage-exclusions.md — Generated and runner-scope filters
    ".features-gen/**",
    "tests/demo-captures/**",
    "**/.claude/**",
  ],
  coverage: {
    // adr: docs/adr/adr-coverage-exclusions.md — Barrel-file re-exports (no logic)
    exclude: ["src/**/index.ts"],
  },
}
```

## Consequences

### Positive

- **Gate signal stays meaningful.** Threshold violations represent
  real regressions on code that *can* be unit-tested.
- **Single source of truth.** Future maintainers hitting an unexpected
  exclusion can grep for the ADR reference and understand the
  rationale immediately.
- **Promotion path is explicit.** If a BDD-only handler grows a
  unit-testable helper module, that module *can* and *should*
  graduate out of the exclusion — this ADR doesn't grant blanket
  immunity to the surrounding crate.

### Negative / risks

- **Coverage of these paths is invisible to the rust matrix.** A
  regression in handler logic that the BDD harness doesn't catch
  would slip past every coverage-driven gate. Mitigation: the
  `api-smoke`, `demo-smoke`, `desktop-e2e`, and `kikan-invariants`
  jobs cover these surfaces from a different angle.
- **Drift risk.** New crates added to `apps/` or new BDD-only
  handlers added to `mokumo-shop` won't be auto-excluded; someone has
  to update both `crap4rs.toml` and `moon.yml`. Quarterly grep audit
  catches missing references.

## Out of scope

- **Cucumber `@wip` and `@future` tags** in `tools/bdd-lint`. Each
  tagged scenario should have its own tracking issue, not an ADR
  reference. Audit owned by #696.
- **Test exclusions added by individual PRs** (e.g., a temporarily-
  flaky test). Those follow the issue-tracking path, not the ADR
  path.

## When this ADR should be revisited

- Cucumber-rs gains nextest compatibility or LCOV emission → most
  BDD-only paths can re-enter the coverage gate.
- A `kikan-admin-ui` build-artifact pre-step lands in the coverage
  workflow → that crate can drop out of the `--exclude` list.
